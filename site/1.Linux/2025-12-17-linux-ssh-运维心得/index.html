    起因¶

    在一个风和日丽的下午，我们正常上的班，我的后端同事给我发消息说为什么我的容器无法启动了呢，其他环境都是没有问题的，当时出问题的是预生产环境.我当时也没在意，以为是那台主机资源用的差不多了，无法调度了。没有考虑到会有人黑进来。

    安全事件案例¶

    事件概述¶

    时间：2025年7月 影响范围：预生产 集群全部节点 + 生产 集群 xxx 节点 攻击方式：SSH 弱密码暴力破解（原因：同事写了传镜像的小脚本，但是因为他嫌普通用户权限太低，于是将root开启，并配置了弱口令密码） 攻击结果：成功入侵并植入挖矿程序 处置结果：已成功清理并恢复正常运行

    事件时间线¶

    1. 初始入侵阶段¶

    攻击入口：预生产 集群节点 xxx-1

    • 攻击手段：攻击者通过 SSH 暴力破解，成功利用弱密码登录到 xxx-1 节点
    • 入侵方式：直接通过 SSH 密码认证方式获得系统访问权限
    • 初始影响：单节点被入侵，攻击者获得普通用户或 root 权限

    2. 恶意程序植入阶段¶

    植入内容：挖矿程序（Cryptocurrency Mining Malware）

    • 程序特征：
    • 占用大量 CPU/GPU 资源进行加密货币挖矿
    • 可能包含持久化机制（systemd 服务、crontab、启动脚本等）
    • 可能包含横向移动工具（SSH 密钥、密码扫描工具等）
    • 检测迹象：
    • 系统资源异常占用（CPU/GPU 使用率飙升）
    • 容器调度失败（资源不足导致）
    • 网络流量异常（与挖矿池通信）

    3. 横向扩散阶段¶

    扩散原因：集群内多个节点共用弱密码

    • 扩散路径：
    • 攻击者在 xxx-1 节点上获取到集群内其他节点的密码信息
    • 利用相同的弱密码，通过 SSH 批量登录到 预生产 集群的其他节点
    • 进一步扩散到 生产 集群的 xxx 节点
    • 扩散速度：由于密码复用，攻击在短时间内扩散至整个集群
    • 影响范围：
    • 预生产 集群：全部节点受影响
    • 生产 集群：xxx 节点受影响

    4. 事件发现阶段¶

    发现方式：业务异常触发告警

    • 告警异常： CPU 使用率远高于100%
    • 触发点：后端同事反馈预生产环境容器无法启动
    • 初步判断：误以为是资源不足导致的调度问题
    • 实际情况：挖矿程序占用大量资源，导致容器无法正常调度

    事件处置过程¶

    1. 应急响应¶

    立即行动： - 隔离受影响节点，防止进一步扩散 - 切断攻击者的持续访问通道 - 收集攻击痕迹和日志证据

    2. 恶意程序清理¶

    清理步骤： 1. 进程清理： - 识别并终止所有挖矿相关进程 - 检查异常进程和网络连接 - 清理僵尸进程和异常子进程

    1. 文件清理：
    2. 删除挖矿程序二进制文件
    3. 清理临时目录和缓存中的恶意文件

    4. 检查并清理 /tmp、/var/tmp 等目录

    5. 持久化机制清理：

    6. 检查并清理 systemd 服务文件
    7. 清理 crontab 定时任务
    8. 检查并清理启动脚本（/etc/rc.local、~/.bashrc、~/.profile 等）

    9. 清理 SSH authorized_keys 中的未授权公钥

    10. 网络清理：

    11. 阻断与挖矿池的网络连接
    12. 清理异常网络规则和防火墙配置

    3. 凭证重置¶

    重置范围：所有受影响节点

    重置内容： - 用户密码重置： - 重置所有用户的密码 - 清理弱密码账户 - 强制使用强密码策略

    • SSH 密钥清理：
    • 清理所有未授权的 SSH 公钥
    • 重置 root 用户的 authorized_keys

    • 检查并清理其他用户的 authorized_keys

    • 服务账户凭证：

    • 重置服务账户密码
    • 更新 API 密钥和访问令牌

    4. 安全策略强化¶

    实施的改进措施：

    1. 密码策略强化：
    2. 实施强密码策略（最小长度、复杂度要求）
    3. 禁止密码复用

    4. 禁用或锁定未使用的账户

    5. SSH 安全加固：

    6. 禁用密码认证，强制使用 SSH 密钥认证
    7. 配置 AllowUsers 限制允许登录的用户
    8. 禁止 root 用户直接登录

    9. 启用 SSH 配置防篡改监控

    10. 监控告警增强：

    11. 部署 SSH 安全监控服务（本文档描述的系统）
    12. 配置异常登录告警
    13. 监控系统资源使用情况

    14. 配置挖矿程序特征检测告警

    15. 重装被侵入的系统

      • 通过手动删除是无法完全清理干净黑客留下的挖坑程序，虽然ssh 做了安全监控，但是不清楚他是否留下了其他口子，通过非ssh 方式进来，重装还是更安全一些。

    16. 禁止公网22端口

    17. 禁止了公网的22端口。

    事件根因分析¶

    1. 密码策略不足¶

    问题点： - 本身我们团队的人员都是使用的密钥对，但是团队就一人可能对安全方面认识太少，配置了root 弱口令，让挖坑人员很轻易攻入 - 同事还在系统上配置了空密码，就会导致使用普通用户直接敲回车就可以进入系统，简直离谱。这也是我在后续的测试时候发现的，当时是因为他说他的账号通过IPMI 无需输入密码，直接回车就可以进入系统我觉得很神奇，就和他请教怎么做到的，结果很离谱竟然配置了空密码 - 事实证明没事别瞎搞，这样系统会很安全，如果搞就按照正常的流程走。幸运的是只是挖坑没有做别的。 - 密码复用导致单点突破后全集群沦陷

    影响： - 攻击者一旦破解一个节点，即可访问整个集群 - 弱密码容易被暴力破解工具攻破 - 缺乏密码管理导致安全风险累积

    2. 监控告警缺失¶

    问题点： - 缺乏 SSH 登录异常监控 - 缺乏系统资源使用监控 - 缺乏恶意程序特征检测 - 告警机制不完善，事件发现滞后

    影响： - 攻击行为未被及时发现 - 挖矿程序运行多时才被发现 - 无法快速定位攻击来源和影响范围

    3. SSH 安全配置不足¶

    问题点： - 允许密码认证 - 未限制允许登录的用户 - 可能存在 root 用户直接登录 - 缺乏 SSH 配置防篡改机制

    影响： - 攻击者可以通过暴力破解获得访问权限 - 配置可能被攻击者修改以维持访问 - 缺乏配置变更监控

    经验教训与改进措施¶

    1. 密码管理¶

    必须实施： - ✅ 强制使用强密码策略 - ✅ 禁止密码复用 - ✅ 定期更换密码 - ✅ 使用密码管理工具 - ✅ 实施多因素认证（MFA）

    2. SSH 安全¶

    必须实施： - ✅ 禁用密码认证，强制密钥认证 - ✅ 配置用户白名单（AllowUsers） - ✅ 禁止 root 直接登录 - ✅ 启用 SSH 配置防篡改监控

    3. 监控告警¶

    必须实施： - ✅ 部署 SSH 安全监控服务 - ✅ 监控异常登录行为 - ✅ 监控系统资源使用 - ✅ 配置恶意程序特征检测 - ✅ 建立完善的告警机制

    4. 应急响应¶

    必须建立： - ✅ 应急响应流程 - ✅ 事件处置预案 - ✅ 定期演练和培训 - ✅ 事件复盘和改进机制

    下面是一个小程序，主要是来实时监控ssh服务是否异常，ssh配置是否发生篡改，是否注入异常公钥！并且按照我们的规则来清理密码等来加强系统监控。

    系统概述¶

    SSH 安全配置和监控服务是一个 Kubernetes DaemonSet，运行在所有节点上，负责：

    1. SSH 配置管理：自动应用和维护安全的 SSH 配置
    2. 配置防篡改：监控 SSH 配置文件，检测并自动恢复篡改
    3. 用户密码安全：监控和清理用户密码，确保密码安全策略
    4. Root 公钥检测：检测并清理 root 用户的未授权 SSH 公钥
    5. Prometheus 监控：生成安全指标，支持告警

    组件架构¶

    Kubernetes 资源¶

    ssh-security-pod (DaemonSet)
├── ConfigMap: ssh-config-template
│   ├── ssh_config.yaml                    # SSH 配置模板
│   ├── monitor-ssh-config.sh              # 主监控脚本（入口）
│   ├── configure-ssh-flexible.sh          # SSH 配置应用脚本
│   ├── user_password_status.sh            # 用户密码状态检查
│   ├── clear_user_passwd.sh               # 用户密码清理
│   ├── check_root_authorized_keys.sh       # Root 公钥检测
│   └── clear_root_authorized_keys.sh       # Root 公钥清理
└── ConfigMap: ssh-config-per-nodes (可选)
    └── ssh_config_per_nodes.yaml          # 节点特定配置

    容器挂载¶

    • /host/etc → 宿主机 /etc（SSH 配置）
    • /host/var → 宿主机 /var（日志、指标）
    • /host/root → 宿主机 /root（root 公钥检测）
    • /scripts → ConfigMap 脚本（可执行）
    • /config → ConfigMap 配置模板
    • /config-nodes → 节点特定配置（可选）

    Kubernetes DaemonSet 启动

容器启动（privileged=true, hostNetwork=true）
    ↓
执行入口命令: /scripts/monitor-ssh-config.sh

    2. 初始化阶段¶

    monitor-ssh-config.sh 启动
    ↓
[1] 显示启动横幅
    - 节点名称
    - 当前时间
    - 指标文件路径
    ↓
[2] 执行初始 SSH 配置
    └─→ configure-ssh-flexible.sh
        ├─ 备份原始配置
        ├─ 应用全局配置（ssh_config.yaml）
        ├─ 应用节点特定配置（ssh_config_per_nodes.yaml，如果存在）
        ├─ 验证配置语法

[4] 进入监控循环（MONITOR_MODE=true）

    3. 监控循环阶段¶

    每 CHECK_INTERVAL 秒（默认 60 秒）执行一次：

[1] 检查配置模板是否变化
    - 计算 ConfigMap 模板的 MD5
    - 如果变化 → 触发配置更新

[2] 检查宿主机配置是否被篡改
    - 计算宿主机配置的 MD5
    - 如果变化 → 记录篡改，触发恢复

[3] 如果检测到变化或篡改
    └─→ configure-ssh-flexible.sh（恢复配置）

[4] 正常检查流程（静默执行）
    ├─→ user_password_status.sh
    │   └─ 生成用户密码状态指标（静默）
    │
    ├─→ clear_user_passwd.sh
    │   └─ 清理用户密码（仅在发生操作时输出日志）
    │
    ├─→ check_root_authorized_keys.sh
    │   └─ 检查 root 公钥（仅在检测到时输出告警）
    │
    └─→ clear_root_authorized_keys.sh（如果检测到公钥）
        └─ 清理 root 公钥并生成指标

[5] 更新 Prometheus 指标
    - 篡改计数和时间戳
    - root 公钥清理状态

    脚本功能说明¶

    1. monitor-ssh-config.sh (主监控脚本)¶

    功能： - 主入口脚本，负责整体监控流程 - 管理 SSH 配置防篡改 - 协调各个子脚本的执行 - 生成 Prometheus 指标

    关键变量： - tamper_recent: 最近篡改次数（1 小时稳定后清零） - last_tamper_time: 最后篡改时间戳 - RECOVERY_STABLE_THRESHOLD: 3600 秒（1 小时）

    执行流程： 1. 初始化阶段：执行一次 SSH 配置 2. 监控循环：每 60 秒检查一次 3. 静默运行：正常情况不输出日志，仅在异常时输出

    2. configure-ssh-flexible.sh (SSH 配置应用)¶

    功能： - 解析 YAML 配置模板 - 应用全局默认配置 - 应用节点特定配置（覆盖全局） - 验证配置语法 - 重新加载 SSH 服务

    配置来源： - /config/ssh_config.yaml - 全局配置 - /config-nodes/ssh_config_per_nodes.yaml - 节点特定配置（可选）

    执行时机： - 容器启动时（初始化） - 检测到配置模板变化时 - 检测到宿主机配置被篡改时

    关键操作： - 备份原始配置到 /host/etc/ssh/backup/ - 使用 nsenter 在宿主机命名空间内操作 - 验证语法：sshd -t - 重新加载服务：systemctl reload sshd

    3. user_password_status.sh (用户密码状态检查)¶

    功能： - 读取 /host/etc/shadow 文件 - 生成用户密码状态指标 - 状态值： - 0: 密码为空 - 1: 密码已设置 - 2: 密码已锁定（! 或 * 开头）

    输出： - Prometheus 指标文件：/host/var/lib/node_exporter/textfile/user_password_status.prom - 静默执行，不输出日志

    执行时机： - 监控循环中每 60 秒执行一次

    4. clear_user_passwd.sh (用户密码清理)¶

    功能： - 扫描所有用户密码 - 清理已设置密码的用户（跳过 openbayes 用户） - 更新锁定密码格式为 !* - 锁定空密码账户

    处理逻辑： 1. 已设置密码 → 清理密码并锁定 

    passwd -d $user
usermod -p '*' $user
passwd -l $user

    1. 锁定格式异常 → 更新为 !* 

      usermod -p '!*' $user

    2. 空密码 → 锁定账户 

      usermod -p '!*' $user

    输出标记： - CLEAR_PASSWORD:<img src="https://latex.codecogs.com/gif.latex?user%20-%20清理密码-%20UPDATE_LOCKED:"/>user - 更新锁定 - LOCK_EMPTY:<img src="https://latex.codecogs.com/gif.latex?user%20-%20锁定空密码日志：-%20文件：/host/var/log/clearuserpass.log-%20仅在发生操作时输出到主脚本日志---###%205.%20check_root_authorized_keys.sh%20(Root%20公钥检测)功能：-%20检查%20/host/root/.ssh/authorized_keys%20文件-%20如果文件存在且非空，输出检测标记输出：-%20ROOT_KEYS_DETECTED:"/>file_path - 检测到公钥文件

    执行时机： - 监控循环中每 60 秒执行一次 - 仅在检测到公钥时触发清理

    6. clear_root_authorized_keys.sh (Root 公钥清理)¶

    功能： - 清理 root 用户的 SSH 授权公钥文件 - 检测并移除不可变标志（chattr +i） - 生成 Prometheus 告警指标

    处理流程： 1. 检查文件是否存在且非空 2. 使用 nsenter 在宿主机命名空间内操作： - 检查并移除不可变标志（chattr -i） - 设置文件权限（chmod 644） - 清空文件内容（: > file） - 如果清空失败，删除并重新创建 3. 生成 Prometheus 指标： - root_keys_clear_failed_detected - 清理是否失败 - root_keys_immutable_detected - 是否检测到不可变标志

    输出标记： - CLEAR_ROOT_KEYS:<img src="https://latex.codecogs.com/gif.latex?file_path%20-%20清理成功-%20ERROR:"/>file_path - 清理失败 - WARNING_IMMUTABLE:$file_path - 检测到不可变标志 - TOTAL_CLEARED:1 - 清理成功 - TOTAL_FAILED:1 - 清理失败 - IMMUTABLE_DETECTED:1 - 检测到不可变标志

    日志： - 文件：/host/var/log/clearrootkeys.log

    监控流程¶

    正常监控流程（静默运行）¶

    每 60 秒循环：
    ↓
[1] 检查配置模板 MD5
    ├─ 未变化 → 继续
    └─ 变化 → 触发配置更新
    ↓
[2] 检查宿主机配置 MD5
    ├─ 未变化 → 继续
    └─ 变化 → 记录篡改，触发恢复
    ↓
[3] 执行用户密码检查（静默）
    ├─ user_password_status.sh → 生成指标
    └─ clear_user_passwd.sh → 清理密码（如有需要）
    ↓
[4] 执行 root 公钥检查
    ├─ check_root_authorized_keys.sh
    ├─ 未检测到 → 重置指标为 0
    └─ 检测到 → 执行清理并输出告警
    ↓
[5] 更新 Prometheus 指标
    ↓
[6] 判断是否稳定恢复
    ├─ 距离最后篡改 > 1 小时 → 重置篡改计数
    └─ 否则 → 保持计数

    异常处理流程¶

    检测到 SSH 配置被篡改：
    ↓
[1] 输出告警日志
    - ⚠️ 检测到宿主机 SSH 配置被篡改！
    - 显示期望和当前 MD5
    ↓
[2] 更新篡改计数
    - tamper_recent++
    - last_tamper_time = 当前时间戳
    ↓
[3] 更新 Prometheus 指标
    ↓
[4] 自动恢复配置
    └─→ configure-ssh-flexible.sh
        ├─ 应用配置
        ├─ 验证语法
        └─ 重新加载服务
    ↓
[5] 更新期望 MD5
    ↓
[6] 继续监控

    配置说明¶

    SSH 配置模板 (ssh_config.yaml)¶

    位置：ConfigMap ssh-config-template → /config/ssh_config.yaml

    格式： 

    # 键值对格式
AllowUsers: "user1 user2 user3"
PermitRootLogin: "no"
PasswordAuthentication: "no"

    支持的配置项： - AllowUsers - 允许的用户列表 - PermitRootLogin - 是否允许 root 登录 - PubkeyAuthentication - 是否启用公钥认证 - PasswordAuthentication - 是否启用密码认证 - StrictModes - 严格模式 - ClientAliveInterval - 客户端保活间隔 - 其他 SSH 配置项...

    节点特定配置 (ssh_config_per_nodes.yaml)¶

    位置：ConfigMap ssh-config-per-nodes → /config-nodes/ssh_config_per_nodes.yaml

    格式： 

    node-name-1:
  AllowUsers: "user1 user2"
  PasswordAuthentication: "yes"

node-name-2:
  AllowUsers: "user3 user4"
  PermitRootLogin: "yes"

    说明： - 节点特定配置会覆盖全局配置 - 如果节点不在配置中，使用全局默认配置

    环境变量¶

    变量名 默认值 说明
    NODE_NAME (自动) 节点名称（从 Kubernetes 获取）
    MONITOR_MODE true 是否启用监控模式
    CHECK_INTERVAL 60 检查间隔（秒）

    Prometheus 指标¶

    SSH 配置防篡改指标¶

    文件：/host/var/lib/node_exporter/textfile/ssh_security.prom

    指标名 类型 说明
    ssh_config_tamper_recent gauge 最近篡改次数（1 小时稳定后清零）
    ssh_config_last_tamper_timestamp gauge 最后篡改时间戳（Unix 时间戳）

    标签： - node: 节点名称

    用户密码状态指标¶

    文件：/host/var/lib/node_exporter/textfile/user_password_status.prom

    指标名 类型 说明
    user_password_status gauge 用户密码状态（0=空, 1=已设置, 2=已锁定）

    标签： - user: 用户名 - host: 主机名

    Root 公钥清理指标¶

    文件：/host/var/lib/node_exporter/textfile/root_keys_clear_failed.prom

    指标名 类型 说明
    root_keys_clear_failed_detected gauge Root 公钥清理是否失败（1=失败, 0=成功）
    root_keys_immutable_detected gauge 是否检测到不可变标志（1=是, 0=否）

    标签： - node: 节点名称

    Prometheus 告警规则¶

    位置：helmfile/common/values/prometheus-rules.yaml

    告警规则： 1. SSHConfigTampered - SSH 配置被篡改 2. SSHConfigFrequentTampering - SSH 配置频繁被篡改（1 小时内 ≥ 3 次） 3. RootKeysClearFailed - Root 公钥清理失败 4. RootKeysImmutableDetected - 检测到 Root 公钥文件被设置为不可变

    故障排查¶

    1. 容器无法启动¶

    检查项： - ConfigMap 是否存在 - 脚本权限是否正确（0755） - 容器镜像是否拉取成功

    2. SSH 配置失败¶

    检查项： - 查看容器日志：kubectl logs -n infra ssh-security-pod-xxx - 检查配置语法：sshd -t -f /etc/ssh/sshd_config - 检查备份目录：/host/etc/ssh/backup/

    3. 监控不工作¶

    检查项： - 环境变量 MONITOR_MODE 是否为 true - 检查间隔 CHECK_INTERVAL 是否合理 - 查看主脚本日志输出

    4. 指标未生成¶

    检查项： - 指标目录是否存在：/host/var/lib/node_exporter/textfile/ - 目录权限是否正确 - node-exporter 是否在收集该目录

    注意事项¶

    1. 权限要求：
    2. 容器需要 privileged=true 和 hostNetwork=true

    3. 需要挂载宿主机文件系统

    4. 性能影响：

    5. 默认检查间隔为 60 秒

    6. 正常情况静默运行，不影响性能

    7. 配置更新：

    8. 修改 ConfigMap 后，Pod 会自动检测并应用新配置

    9. 节点特定配置会覆盖全局配置

    10. 安全考虑：

    11. 所有操作都通过 nsenter 在宿主机命名空间内执行
    12. 配置备份保存在 /etc/ssh/backup/
    13. 篡改检测基于 MD5 校验和
